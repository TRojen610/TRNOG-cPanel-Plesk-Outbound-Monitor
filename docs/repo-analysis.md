# TRNOG Hosting Outbound Logger Repository Analysis

## Scope

- Upstream repository: `https://github.com/TRNOG/hosting-outbound-logger`
- Audited clone path: `repo/hosting-outbound-logger`
- Audited branch: `main`
- Audited HEAD during analysis: `ebd16df`

## Purpose Of The Original Project

The upstream project is an eBPF/BCC-based outbound connection logger for Linux hosting servers. Its goal is to identify which local process and user opened an outbound TCP connection, enrich that event with optional hostname data, and emit the result as JSON or CSV for local file logging and/or remote SIEM forwarding.

The project is positioned for shared hosting and general server monitoring environments such as cPanel, Plesk, DirectAdmin, and plain Linux. It does not capture traffic payloads. It only logs metadata about outbound connections.

## Repository Layout

The repository is intentionally small. The current `main` branch contains only three tracked files:

- `README.md`
  Describes the product goals, supported systems, install flow, logging fields, and service management model.
- `install.sh`
  The real application source. This script installs dependencies, asks runtime questions, writes configuration, generates the Python logger code, writes rsyslog rules, writes a systemd unit, and supports uninstall.
- `LICENSE`
  MIT license.

There are no standalone Python modules, no packaged service code, no tests, and no web or cPanel integration assets in the upstream repository. The functional runtime is created dynamically by `install.sh`.

## Architecture Summary

The upstream design is a shell installer that generates a Python eBPF logger and wires it into the host OS.

High-level architecture:

1. `install.sh` detects the package manager and installs Python, BCC/BPFCC, kernel headers, and rsyslog.
2. `install.sh` asks the operator how logs should be emitted:
   - file
   - syslog
   - both
3. `install.sh` writes `/etc/outbound-logger/outbound-logger.env`.
4. `install.sh` generates `/opt/outbound-logger/logger.py`.
5. `install.sh` writes rsyslog rules under `/etc/rsyslog.d/`.
6. `install.sh` writes `/etc/systemd/system/outbound-logger.service`.
7. systemd starts the Python process as `root`.
8. The Python process loads a BPF program through BCC, attaches kernel probes to outbound TCP connect paths, optionally attaches user-space probes for DNS and TLS SNI hostname hints, and prints records to stdout.
9. systemd sends stdout to journald; rsyslog filters those records and writes them to daily files and/or forwards them to a remote syslog receiver.

## Important Runtime Files Generated By The Installer

These files are not tracked in Git, but they are central to the real runtime model:

- `/opt/outbound-logger/logger.py`
  Generated Python collector containing the BPF program, perf-buffer handlers, record formatting, filtering, and hostname enrichment logic.
- `/etc/outbound-logger/outbound-logger.env`
  Environment file storing output mode, log format, and ignore/enrichment switches.
- `/etc/systemd/system/outbound-logger.service`
  Service unit that runs the generated Python logger as `root`.
- `/etc/rsyslog.d/29-outbound-logger-template.conf`
  Optional dynafile template used for daily file naming.
- `/etc/rsyslog.d/30-outbound-logger.conf`
  rsyslog routing rule for local file and/or remote forwarding.
- `/var/log/outbound-logger/YYYY-MM-DD.jsonl`
  Default local log location when file logging is enabled.

## Data Flow

1. A local process opens an outbound TCP connection.
2. The BPF program hooks:
   - `tcp_v4_connect`
   - `tcp_v6_connect`
3. BCC delivers IPv4 or IPv6 event structs to Python via perf buffers.
4. Python resolves:
   - UID to username
   - PID to `/proc/<pid>/cmdline`
   - source and destination IP/port
   - process name from `comm`
5. Optional hostname enrichment is populated from:
   - `getaddrinfo()` uprobe on libc
   - `SSL_set_tlsext_host_name()` uprobe on libssl
6. Python applies runtime filters:
   - ignore loopback
   - ignore private destinations
   - ignore root-owned processes
7. Python emits one line per event to stdout:
   - JSON line when `LOG_FORMAT=json`
   - quoted CSV line when `LOG_FORMAT=csv`
8. systemd/journald captures stdout.
9. rsyslog matches the journal message and:
   - writes it to a daily file
   - forwards it to remote syslog
   - or both

## How Outbound Events Are Collected

The generated Python logger uses BCC and an embedded BPF program.

Collection mechanics:

- `trace_connect_v4_entry` and `trace_connect_v6_entry` store the current socket pointer in a `BPF_HASH`.
- `trace_connect_v4_return` and `trace_connect_v6_return` read socket metadata on successful connect and submit event structs through perf buffers.
- IPv4 and IPv6 events include:
  - timestamp in microseconds
  - pid
  - uid
  - source address
  - destination address
  - source port
  - destination port
  - process name (`comm`)
- Optional host enrichment uses perf events from user-space probes on:
  - `getaddrinfo`
  - `SSL_set_tlsext_host_name`

## Log Format And Storage Strategy

Structured fields emitted by the Python logger:

- `timestamp`
- `hostname`
- `uid`
- `user`
- `pid`
- `comm`
- `cmdline`
- `family`
- `src_ip`
- `src_port`
- `dst_ip`
- `dst_port`
- optional `dst_host`
- optional `dst_host_source`

Storage strategy:

- The collector does not write log files directly.
- It prints formatted records to stdout.
- systemd routes stdout to journald.
- rsyslog filters journal messages and writes daily files or forwards remotely.

Default daily file pattern:

- `/var/log/outbound-logger/YYYY-MM-DD.jsonl`

## Dependencies

Directly implied by the installer and generated code:

- `python3`
- `python3-pip`
- BCC/BPFCC Python bindings:
  - `python3-bpfcc` on Debian/Ubuntu
  - `python3-bcc` on RHEL-family systems
- BCC tools:
  - `bpfcc-tools` or `bcc-tools`
- `rsyslog`
- matching kernel development packages:
  - `linux-headers-$(uname -r)` on apt-based systems
  - `kernel-devel` and `kernel-headers` on dnf/yum systems
- a system with:
  - systemd
  - journald
  - rsyslog
  - root access
  - a kernel/new-enough BPF stack compatible with BCC

Host enrichment additionally assumes:

- libc exposes `getaddrinfo`
- libssl exists in one of several hard-coded common paths or is discoverable through `ldconfig`

## OS-Level And Service Assumptions

- Linux only
- root execution is required
- package manager must be one of:
  - `apt`
  - `dnf`
  - `yum`
- service management assumes `systemctl`
- log routing assumes rsyslog is present and active
- filesystem paths are hard-coded for a traditional Linux host layout
- OpenSSL shared library paths are assumed to match common x86_64 distributions
- The collector is built around outbound TCP connect hooks only

## Service Model

The original project is not an HTTP service.

It is a single long-running Python process managed by systemd. That process emits structured lines to stdout. File and remote forwarding are delegated to rsyslog rather than being handled in-process.

This means the upstream repository currently provides:

- collection
- formatting
- OS install automation
- journald/rsyslog integration

It does not provide:

- API endpoints
- state management
- web GUI
- browser access
- cPanel/WHM embedding
- auth model for a separate service port

## Risks And Limitations

### Current design risks

- The runtime application code is generated inside `install.sh` instead of being versioned as normal source files. That makes review, testing, packaging, and open-source collaboration harder.
- Installation is interactive only. There is no clean non-interactive configuration mode for automation.
- Package installation commands frequently end with `|| true`, so dependency installation can fail silently and only surface later at runtime.
- The runtime depends on BCC and kernel headers matching the running kernel, which is a common operational failure point on hosting nodes.
- The logger only captures outbound TCP connect activity. UDP and other protocols are not implemented.
- Hostname enrichment is best-effort and keyed only by PID with a short TTL, so hostname association can be approximate.
- The service runs as `root` with BPF access and probes libc/libssl globally.
- There is no retention policy, compaction, or local search/indexing layer.
- There is no HTTP/API security model because there is no HTTP layer yet.

### Concrete implementation issue found during audit

- The rsyslog filter in `install.sh` matches messages containing JSON field names such as `"src_ip"` and `"cmdline"`.
- In `csv` mode, the generated logger emits quoted CSV rows instead of JSON.
- Result: the current rsyslog routing rule will not match CSV output, so CSV mode is not reliable for file/syslog delivery without fixing the routing logic.

### Integration limitations for cPanel/WHM

- The original project has no concept of tenancy, authorization, or panel session handling.
- Output is journal/file/syslog oriented, not API oriented.
- There is no browser-facing process or reverse-proxy integration.
- There is no packaging layout for `/usr/local/cpanel` or WHM plugin registration.

## What Can Be Reused Directly

The most valuable reusable pieces are:

- The embedded BPF program and its probe strategy
- The event schema produced by the generated Python logger
- The Python-side metadata enrichment:
  - username lookup
  - cmdline lookup
  - loopback/private/root filters
  - hostname cache and best-effort DNS/SNI attribution
- The installer’s knowledge of required system packages
- The existing file naming convention and default log path
- The general service identity and naming:
  - `outbound-logger`

## What Needs Wrappers, Adapters, Or Refactoring

To become a production-grade open-source cPanel/WHM module with a standalone service, the following changes are required:

- Extract the generated Python code into versioned source files under `core/`
- Replace interactive-only configuration with a real config file and scriptable install flow
- Add a long-running backend service under `service/` with:
  - structured state management
  - HTTP API
  - health checks
  - realtime stream delivery
- Normalize event output into an internal model independent of stdout/journald parsing
- Keep file export compatibility, but stop relying exclusively on rsyslog for application state
- Build a web UI under `web/`
- Add WHM/cPanel integration files under `cpanel/`
- Add installer/uninstaller scripts that deploy both the collector and the web service
- Add documentation and packaging suitable for open-source collaboration

## Recommended Refactoring Direction

The lowest-risk path is to preserve the logger logic but change how it is hosted:

1. Move the generated Python collector into source-controlled Python modules.
2. Keep the BPF probes and record schema as close to upstream as possible.
3. Wrap the collector in a backend service that receives structured events directly instead of re-parsing journal output.
4. Optionally keep file export compatibility so the original operational model remains familiar.
5. Expose the same event stream through an internal API and a WHM-facing GUI.

This approach preserves the core project while making it maintainable and extensible enough for public TRNOG release.
